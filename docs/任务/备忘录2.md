这个问题问得非常精准。现在我对两个项目都有了深入了解，关键发现是：**waoowaoo 的强项和弱项，恰好与你的三个场景形成非常有趣的互补关系。**

---

## 核心发现：waoowaoo 是单人产品，我们是多人平台

先摆一个关键事实：

| 维度 | waoowaoo | 我们的项目 |
|------|----------|-----------|
| **审核流** | 无（零审核机制） | 完整（human/ai/human+ai 三模式，6 种审核状态，自动重生成） |
| **权限模型** | 无（project 只属于一个 user） | 完整（RBAC + 7 种工种角色 + 资源状态规则） |
| **多人协作** | 无（无 team/member/role） | 完整（项目成员、团队、阶段锁、镜头任务锁、乐观锁） |
| **工程守卫** | 极强（24 个守卫脚本） | 弱（仅 Makefile + 手动脚本） |
| **测试契约** | 极强（路由目录 + 行为矩阵 + 5 个覆盖守卫） | 中等（21 个后端测试，无前端覆盖守卫） |
| **AI 运行时** | 完整（run/step/event/seq） | 分散（Asynq + WS + 各模块独立） |

结论很清楚：**在你的三个特殊场景下，我们不需要照搬它的全部，而是精准吸收与我们场景匹配的部分。**

## 场景一：AI 驱动开发 → 最需要工程守卫

这是 waoowaoo 对我们**价值最大、最紧急**的部分。

AI 写代码最大的风险不是"写不出来"，而是**架构回归**——AI 不记得上次遵守了什么约束，每次对话都可能违反已建立的规则。你在 `.cursor/rules/` 里写了详细的规范，但这些规则只是"建议"，不是"机器强制"。

waoowaoo 的 24 个守卫脚本，本质上是把"写在文档里的规范"变成了"机器可执行的检查"。以下是**直接适用于我们项目的守卫类型**：

### 应该立刻建立的守卫

**1. 架构边界守卫**（防止 AI 绕过分层）

waoowaoo 有 `no-api-direct-llm-call`（禁止 API 层直接调 LLM），我们应该做：

```bash
# 思路：扫描 handler 层，禁止直接 import service 以外的数据层
# 禁止 handler 直接 import data_store / queries
# 禁止 service 直接 import gin.Context
```

这防止 AI 写代码时走捷径：比如在 handler 里直接调 sqlc queries，绕过 service 层。

**2. 文件行数守卫**（我们规则里已写"600 行以内"，但没有机器检查）

waoowaoo 的 `file-line-count-guard.mjs` 限制 component 500 行、hook 400 行。我们已经在 `ai-development-guidelines.mdc` 里规定了 600 行限制，但只靠 AI 自觉。应该变成脚本：

```bash
# 扫描 anime_ai/**/*.go 和 anime_ui/lib/**/*.dart
# 排除 sch/db/*.sql.go（sqlc 生成代码）
# 超过 600 行的文件报错
```

**3. 路由登记守卫**（防止 AI 加了 API 没登记/没测试）

waoowaoo 的 `test-route-coverage-guard.mjs` 对比实际路由与 `route-catalog.ts`。我们应该做：

```bash
# 扫描 route.go 中注册的路由
# 对比 *_test.go 中覆盖的路由
# 新增路由没有对应测试 → 报错
```

**4. 多源真相守卫**（防止 AI 在前端创建 shadow state）

waoowaoo 的 `no-multiple-sources-of-truth.mjs` 特别适合我们。AI 写 Flutter 代码时特别容易犯的错误：把服务端数据复制到 `useState` / `StateNotifier` 本地变量，导致状态不同步。

**5. 测试质量守卫**（防止 AI 写"假测试"）

waoowaoo 的 `test-behavior-quality-guard.mjs` 禁止只有 `toHaveBeenCalled()` 而没有具体参数断言。AI 写测试时最常见的问题就是"结构正确但断言太弱"——看起来通过了，实际什么都没验证。

### 为什么这对 AI 开发特别重要？

因为 AI 有两个天然弱点：
1. **无记忆**：每次对话不记得之前的架构约定，守卫脚本是跨对话的持久记忆
2. **讨好倾向**：AI 倾向于用最短路径完成任务，可能绕过分层直接硬编码——守卫阻止这种捷径

## 场景二：多人协作 → waoowaoo 无参考价值，但 SSE 恢复有用

waoowaoo 是**单用户产品**（Project 只有一个 userId，无 team/member/role），在多人协作上没有任何值得参考的设计。我们的 RBAC + 7 种工种 + 阶段锁 + 镜头锁 + 乐观锁已经远超它。

但有一个间接相关的点：**多人同时查看任务进度时的状态同步**。

当导演和分镜师同时在看同一个项目的镜图生成进度时：
- 导演切了个标签页回来，WebSocket 重连了，但中间的 `task_progress` 事件丢了
- 分镜师网络抖了一下，恢复后不知道哪些镜头已经生成完了

waoowaoo 的 SSE 恢复机制（`last-event-id` + 快照）在这个场景下有价值。但**实现方式应该适配我们的 WebSocket 架构**：

### 建议的实现方式

不需要照搬它的 SSE + `seq` 协议，而是在我们现有的 WebSocket 上加两个能力：

1. **重连快照**：WebSocket 重连时，后端自动推送该项目当前所有活跃任务的最新状态
2. **事件缓存**：在 `RealtimeHub` 中对每个项目维护最近 N 条事件的环形缓冲，重连时补发

这比 waoowaoo 的方案简单得多，但覆盖了多人协作最常见的"断流恢复"场景。

## 场景三：审核门禁流程 → waoowaoo 完全没有，我们是领先方

这一点必须明确说：**waoowaoo 没有任何审核/审批机制**。它的 `run/step` 是纯技术执行模型，没有"需要人审核后才能进入下一步"的概念。

而我们已经有：

| 能力 | 实现位置 |
|------|----------|
| 三种审核模式（human/ai/human+ai） | `module/project/review_config.go` |
| 镜图 6 种审核状态 | `module/shot_image/review_flow.go` |
| AI 审核自动重生成 | `module/shot_image/retry.go` |
| 审核记录持久化 | `pub/review_record/` |
| 阶段锁（脚本锁定→才能生图） | `module/project/service.go` |
| 资源状态规则（frozen 只能审核不能编辑） | `pub/auth/rbac.go` |

但 waoowaoo 的 `run/step/event` 模型有一个思路可以**增强我们的审核流可观察性**：

### 建议：给审核节点加入统一事件流

当前我们的审核流转是"API 调用 → 状态更新 → WebSocket 推送"，但推送的事件类型比较粗（`task_progress` / `task_complete`）。可以考虑增加**审核生命周期事件**：

```
review.submitted     → 提交审核
review.ai_started    → AI 审核开始
review.ai_completed  → AI 审核完成（通过/拒绝）
review.human_pending → 等待人工审核
review.approved      → 审核通过
review.rejected      → 审核拒绝
review.regenerating  → 拒绝后自动重生成
```

这样前端可以实时展示审核流的每一步进展，而不只是"生成中"→"完成/失败"两个状态。这对多人协作场景特别有价值——导演可以实时看到"AI 正在审核第 12 张镜图"。

## 最终建议：按场景排优先级

| 优先级 | 要做的事 | 对应场景 | 参考来源 | 工作量 |
|--------|---------|---------|---------|--------|
| **P0** | 建立 5 个核心守卫脚本（架构边界、文件行数、路由覆盖、多源真相、测试质量） | AI 驱动开发 | waoowaoo guards/ | 小（shell 脚本） |
| **P0** | 加 `make guard` 命令 + 开发流程约定"提交前跑 guard" | AI 驱动开发 | waoowaoo package.json | 极小 |
| **P1** | WebSocket 重连时推送活跃任务快照 | 多人协作 | waoowaoo SSE 恢复思想 | 小 |
| **P1** | 审核生命周期事件扩展（review.submitted 等） | 审核门禁 + 多人协作 | waoowaoo event 协议思想 | 小 |
| **P2** | 测试契约体系（路由目录 + 任务类型目录 + 行为矩阵） | AI 驱动开发 | waoowaoo contracts/ | 中 |
| **P3** | 统一 run/step 执行模型 | 未来多步流水线 | waoowaoo run-runtime/ | 大 |
| **不需要** | 权限/协作/审核流设计 | — | 我们已经远超它 | — |
| **不需要** | 模型配置契约 | — | 我们 `pub/capability/` 已有 | — |

一句话总结：**waoowaoo 对我们最大的价值是"工程守卫"体系——它把"AI 最容易犯的架构错误"变成了机器可检测的规则。这正是 AI 驱动开发最需要的防线。而在协作和审核方面，我们是领先方，不需要向它学习。**
