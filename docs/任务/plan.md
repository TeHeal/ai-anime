## 镜图生成全流程重新设计方案

### 当前问题诊断

你截图中的竞品 UI 提供了一个很好的参考——右侧面板有"选择素材（角色/场景/道具）"、"上传参考图"、"提示词"、"立即生成"的分步面板。但我们的系统有一个它没有的优势：**脚本阶段已经为每个镜头生成了画面描述、角色、情绪、音频设计等结构化数据**，这些数据应该被自动利用，而不是让用户从零开始填写。

### 核心设计理念：三阶段渐进式生图

用电影术语来说：**草图 → 定稿 → 交片**。

```
阶段一：提示词工坊 (Prompt Workshop)
  脚本数据 + 风格设定 + 角色/场景素材 → AI 组装提示词 → 人工微调

阶段二：快速验证 (Quick Proof)  
  低成本快速出图 → 缩略图预览 → 批量筛选 → 标记"满意/重试"

阶段三：正式出图 (Final Render)
  选定的镜头 → 高质量生成（抽卡模式可选）→ 候选图挑选 → 审核确认
```

---

### 阶段一：提示词工坊（Prompt Workshop）

**为什么需要这个阶段？**

当前的致命断裂是：脚本中每个镜头已有丰富的 `sceneDescription`（画面描述）、`characterName`、`emotionDescription`、`characterPosition`、`audioDesignText` 等，但镜图生成时只用了一个手动输入的 `globalPrompt`。这就像导演已经写好了分镜稿，美术却不看就自己画。

**UI 呈现建议：**

这个阶段应该是 Tab 1"生成中心"的**核心改造**——在当前的"生成配置"和"任务区"之间，插入一个**提示词预览/编辑面板**。

布局设想：

```
┌─────────────────────────────────────────────────────────┐
│ 生成配置 (折叠式)                                         │
│  [风格模板▼] [生图模型▼] [画面比例▼]                        │
│  项目风格：自动加载 projectDefaultStyle                      │
│  反向提示词：[预填]                                        │
└─────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────┐
│ 提示词工坊  [一键生成全部提示词] [批量应用风格]                │
│ ┌──────────────────────────┬──────────────────────────┐ │

│ │ ┌──────────────────────┐ │ ┌──────────────────────┐ │ │

│ └──────────────────────────┴──────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

**提示词自动组装逻辑（纯前端）：**

```dart
String assemblePrompt(StoryboardShot shot, Style? projectStyle, Character? char, Location? loc) {
  final parts = <String>[];
  // 1. 风格前缀
  if (projectStyle != null) parts.add(projectStyle.description);
  // 2. 画面描述（脚本核心）
  if (shot.prompt.isNotEmpty) parts.add(shot.prompt);
  // 3. 角色特征
  if (char != null) parts.add('角色: ${char.name}, ${char.appearance}');
  // 4. 场景特征
  if (loc != null) parts.add('场景: ${loc.name}, ${loc.description}');
  // 5. 情绪氛围
  if (shot.emotion?.isNotEmpty == true) parts.add('情绪氛围: ${shot.emotion}');
  // 6. 镜头语言
  parts.add('${shot.cameraType ?? ""} ${shot.cameraAngle ?? ""}');
  return parts.where((p) => p.isNotEmpty).join(', ');
}
```

**数据来源：**

- 项目默认风格 → `assetStylesProvider`（已有 `isProjectDefault`）
- 角色信息 → `assetCharactersProvider`，通过 shot 的 `characterName`/`characterId` 匹配
- 场景信息 → `assetLocationsProvider`，通过 shot 的 `sceneId` 关联
- 脚本数据 → `shotsProvider` 中的 `StoryboardShot`（`prompt`, `stylePrompt`, `emotion`, `cameraType` 等已有）

**素材引用卡片：**

每个镜头卡片上方展示自动匹配到的素材缩略图（角色头像、场景参考图），用户可点击添加/替换，这些参考图将作为 `image` 参数传给 Seedream API 实现"图生图"。

### 阶段二：快速验证（Quick Proof）

**为什么需要这个阶段？**

生成 50 个镜头的高质量图，按 Seedream 5.0 Lite 价格，每张约 0.06 元，50 张 = 3 元。但问题不是钱——而是**时间**。如果提示词有问题，全部高质量生成后才发现，浪费的是等待时间和创作节奏。

快速验证 = **低成本快速出图，确认方向没问题后再精修**。

**UI 呈现建议：**

在提示词工坊完成后，出现一个**快速验证面板**——类似电影预审片（dailies）的概念：

```
┌─────────────────────────────────────────────────────────┐
│ 快速验证 (Quick Proof)     已验证 12/50  通过 8  需调整 4   │
│ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐       │
│ │ #1  │ │ #2  │ │ #3  │ │ #4  │ │ #5  │ │ #6  │ ...   │
│ │ 🖼  │ │ 🖼  │ │ ⏳  │ │ 🖼  │ │  —  │ │  —  │       │
│ │ ✅  │ │ ⚠️  │ │ ... │ │ ✅  │ │     │ │     │       │
│ └─────┘ └─────┘ └─────┘ └─────┘ └─────┘ └─────┘       │
│                                                         │
│ [⚡ 生成全部预览] [☑ 生成选中] [✅ 全部通过→正式出图]       │
└─────────────────────────────────────────────────────────┘
```

**实现方式：**

Seedream API 的 `optimize_prompt_options.mode` 虽然 5.0 Lite 不支持 `fast`，但可以通过以下策略实现"快速预览"：
- **降低分辨率**：用 512x288（16:9）而非正式的 1920x1080
- **单张出图**：`outputCount = 1`，不做抽卡
- **简化提示词**：去掉精修后缀

每个镜头缩略图小卡片，点击放大查看，标记 ✅ 通过 / ⚠️ 需调整提示词。"需调整"的镜头直接跳转回提示词工坊对应卡片编辑。

- 生成完成的镜头自动进入"待挑选"状态（抽卡模式）或"待审核"状态（单张模式）

### 审核编辑（Tab 2 改造）

审核页的当前结构（三栏：列表 / 中心预览 / 右侧面板）是好的，但需要**联通功能**：

**中心面板改进：**
- 候选图从后端 `getCandidates` 拉取，而非仅显示主图
- 点击候选图 → 调 `selectCandidate` → 设为正式镜图
- 脚本对照区增加"提示词对照"——显示生成时用的最终提示词

**右侧面板改进：**
- 审核操作联通 `updateImageReview` API
- "重新生成"联通提示词编辑 + `batchGenerate`
- QA 检查从 mock 改为调 `triggerQA` / `getQAResult`
- 批量操作联通 `batchReview`

### 建议实施顺序

1. **提示词组装逻辑** — 这是基础，不涉及 UI，但后续全部依赖
